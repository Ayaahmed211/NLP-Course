01

What NER actually is

Named Entity Recognition (NER) takes a raw sentence and pulls out the words that refer to real-world things, then labels each one with a category. Told "My name is Ahmed Ibrahim, I lived in Monufia governorate, born in 1998," a person immediately understands: Ahmed Ibrahim is a person's name, Monufia is a place, 1998 is a date.

That's exactly what NER automates. You hand the model a dictionary of categories it should look for, and it scans the text and extracts the matching spans — this word is a person, this one is an action/organization, this one is a date, and so on.

02

Two libraries, two routes

This lesson covers NER three ways: with 🤗 Hugging Face Transformers (a BERT model), with spaCy, and with NLTK. Each is demonstrated on the same kind of sentence so the results can be compared directly.

Heaviest / most accurate

Hugging Face + BERT

Uses a transformer model rather than spaCy's lighter pipeline. In the lesson's harder example, BERT correctly caught a distinction that spaCy's small model missed.

Fast / practical default

spaCy

Ships several pretrained sizes — small, medium, large, and a transformer-based variant — all sharing the same API, so switching model size doesn't change your code.

Classic toolkit

NLTK

Builds entities on top of tokenization and POS tagging using ne_chunk, producing a tree structure instead of a flat list.

Reality check

No model is perfect

All three approaches occasionally mislabel an entity or miss one entirely — the lesson deliberately shows these failures, not just the wins.

03

Route 1 — Hugging Face Transformers

Hugging Face's model hub hosts many pretrained NER models — this lesson uses a BERT-based one. Setting it up is just three choices: pick a task, pick a pretrained model, pick its matching tokenizer, then wrap them in a pipeline.

from transformers import pipeline, AutoModelForTokenClassification, AutoTokenizer model_name = "dslim/bert-base-NER" # one of several pretrained NER models model = AutoModelForTokenClassification.from_pretrained(model_name) tokenizer = AutoTokenizer.from_pretrained(model_name) ner_pipeline = pipeline("ner", model=model, tokenizer=tokenizer) ner_pipeline("Sarah went to London.")

Running it on "Sarah went to London" correctly returns Sarah as a PERSON and London as a LOCATION. The Hugging Face site's model page (linked in the pipeline's own documentation) lists every available pretrained model — small, medium, large, and transformer variants — all interchangeable through the same handful of lines of code.

04

Route 2 — spaCy

As in the POS-tagging lesson, load a pretrained pipeline and hand it your text. spaCy automatically fills in doc.ents with every entity it found.

import spacy nlp = spacy.load("en_core_web_sm") doc = nlp("Tesla Inc. made $5 billion dollars.") print(doc.ents) # -> (Tesla Inc., $5 billion dollars) for ent in doc.ents: print(ent.text, ent.label_, spacy.explain(ent.label_))

As with POS tags, unfamiliar entity labels can always be looked up with spacy.explain(), or in spaCy's own documentation, which lists every label and pipeline attribute (like doc.ents returning a tuple of spans) under each pipeline's page.

05

When the model gets it wrong

Given "Michael Bloomberg founded Bloomberg in 1982," spaCy's small model:

EntityspaCy labelCorrect?
Michael BloombergPERSON✓ Correct
BloombergPERSON✗ Should be ORG (the company/network)
1982DATE✓ Correct
Miss — the small spaCy model mislabels the second "Bloomberg" (the company) as a person instead of an organization.

Running the exact same sentence through the Hugging Face BERT model instead, it correctly separates the two: Michael Bloomberg as a person, and the standalone Bloomberg as an organization.

Fix — the larger transformer-based model (BERT, via Hugging Face) resolves the ambiguity that the lighter spaCy model missed. It's a reminder that model size and training data genuinely affect accuracy on ambiguous cases.

You can also train your own custom NER model for a specific domain — for example, gathering text from Twitter, Facebook, etc., labeling the categories you care about (person, organization, location…), and training a model on that labeled data from scratch.

06

Locating entities precisely

Besides the entity text and label, spaCy tells you exactly where each entity starts and ends in the original string, using start_char and end_char:

for ent in doc.ents: print(ent.text, ent.label_, ent.start_char, ent.end_char) # -> Tesla Inc. ORG 0 10 # -> Twitter Inc. ORG 30 41 # -> $1 billion MONEY 45 55

Those indices let you slice the exact substring back out of the original text — e.g. text[0:10] correctly returns "Tesla Inc." (Python slicing stops one index short of the end, so end_char works out neatly).

A closer look at what the model is keying on

Testing "Twitter" without the word "Inc" following it, or lowercasing "Inc" to "inc", makes the model fail to recognize it as a company at all — in the lowercase case it instead reads "Inc" alone as a person's name. This shows the model has effectively learned a pattern like: a capitalized word immediately followed by a capitalized "Inc" tends to indicate an organization — a good illustration of how these statistical patterns can be both powerful and brittle.

07

Manually correcting an entity with spaCy's Span

When spaCy misses or mislabels something, you can force the correct label onto a specific stretch of text using Span, then merge it back into the document's entities with doc.set_ents().

from spacy.tokens import Span # manually tag token indices 5–7 ("Twitter") as ORG new_entity = Span(doc, 5, 7, label="ORG") doc.set_ents([new_entity], default="unmodified") # default="unmodified" keeps every other existing entity as-is print(doc.ents) # -> now includes Twitter tagged as ORG

Re-running the loop over doc.ents afterward confirms the fix: "Twitter" now shows up labeled ORG alongside spaCy's other, correctly-detected entities.

08

Route 3 — NLTK

NLTK builds NER on top of the same two building blocks from the last lesson: tokenizing and POS-tagging. A third function, ne_chunk, groups the tagged tokens into a tree, marking which branches are named entities.

import nltk tokens = nltk.word_tokenize(text) pos_tags = nltk.pos_tag(tokens) tree = nltk.ne_chunk(pos_tags) print(tree)

The result looks like a set of small nested trees. To pull the entities back out as plain (text, label) pairs, walk the tree and check each subtree's label:

entities = [] for subtree in tree: if hasattr(subtree, "label"): if subtree.label() in ["PERSON", "ORGANIZATION", "GPE"]: words = " ".join(leaf[0] for leaf in subtree.leaves()) entities.append((words, subtree.label())) print(entities)

On the example sentence "Steve Jobs, CEO of Apple, [was born in] California," NLTK correctly returns Steve Jobs and Jobs as PERSON, and CEO / Apple as ORGANIZATION — but, like spaCy's small model earlier, it fails to catch California as a location at all.

Pattern — across all three tools in this lesson, person and organization names tend to be caught reliably, while place names are the category most likely to slip through — worth remembering when picking (or fine-tuning) a model for a real project.
09

Wrap-up

NER is conceptually simple — scan the text, find the spans that represent real-world things, label them with a category — but getting it right in practice depends heavily on which model you use and how it was trained. Hugging Face's BERT model, spaCy's pretrained pipelines, and NLTK's chunker all use different mechanics to arrive at the same kind of answer, and comparing their outputs side by side on the same sentence is a good habit for catching a given model's blind spots before relying on it.
